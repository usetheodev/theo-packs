# Plan: theo-packs as Single Source of Truth for Dockerfile Generation

> **Version 1.0** — theo-packs becomes the only path for producing the Dockerfile that the Theo build pipeline consumes. User-supplied Dockerfiles are rejected with a hard fail; the user-Dockerfile-precedence contract documented in v1 is removed entirely. theo-stacks templates are stripped of their hand-maintained Dockerfiles (26 files across 18 templates). Pre-release window with no external users; no backward compatibility. Two coordinated PRs ship together: theo-packs (CLI + contract + tests) and theo-stacks (delete Dockerfiles + update READMEs + smoke test).

## Context

### Decision (this plan)

theo-packs is the **single source of truth** for Dockerfile generation. The Theo build pipeline always invokes `theopacks-generate` and consumes its output. No fallback to user-supplied Dockerfiles, no warn-and-ignore, no `--respect-user-dockerfile` flag — **hard fail with exit code 2 if `<source>/<app-path>/Dockerfile` exists**.

The user has explicitly waived backward compatibility: pre-release, no external users, "fix in FAANG level".

### What exists today (the contract being removed)

`cmd/theopacks-generate/main.go:60-73`:

```go
// Check if user already provided a Dockerfile — user-provided takes precedence.
userDockerfile := filepath.Join(appDir, "Dockerfile")
if _, err := os.Stat(userDockerfile); err == nil {
    fmt.Fprintf(os.Stderr, "[theopacks] User-provided Dockerfile found at %s — skipping generation\n", userDockerfile)
    content, readErr := os.ReadFile(userDockerfile)
    if readErr != nil {
        log.Fatalf("Failed to read user Dockerfile: %v", readErr)
    }
    if err := os.WriteFile(*output, content, 0644); err != nil {
        log.Fatalf("Failed to copy user Dockerfile to %s: %v", *output, err)
    }
    fmt.Println("--- User-provided Dockerfile ---")
    fmt.Print(string(content))
    return
}
```

This precedence is documented as load-bearing in `docs/contracts/theo-packs-cli-contract.md` (the "User-Dockerfile precedence" section). It must be inverted.

`cmd/theopacks-generate/main_test.go:143` houses `TestUserProvidedDockerfileTakesPrecedence` which locks the current behavior. It must become `TestUserProvidedDockerfileIsRejected` asserting the new hard fail.

### Why this is the right fix now

- **Pre-release window.** No external users; no migration cost.
- **The current precedence is the source of every "theo-packs is broken" misdiagnosis** — the dogfood F2/F5 found a buggy template Dockerfile and blamed theo-packs. Removing the path eliminates the entire class of confusion.
- **Single contract surface.** Two paths (user-Dockerfile and generated) means double the test surface, double the docs, double the failure modes. FAANG-level releases have one path.
- **Templates as fixtures, not products.** theo-stacks templates exist to seed user projects with `app.py`, `package.json`, dependencies — never Dockerfiles. The Dockerfile is a build artifact, not source code.

### Validation already in place

PR #19 (`fix/monorepo-contract-validation`) added `TestE2E_MonorepoTurboFromStacks` which already exercises the "no user Dockerfile" path — it deletes `apps/api/Dockerfile` from the upstream copy before generating. After this plan ships, that deletion becomes a no-op (file is gone upstream) and the test path is the only path.

## ADRs

### D1 — Hard fail with exit code 2; no override
**Decision:** When `<source>/<app-path>/Dockerfile` exists, the CLI logs an error to stderr naming the offending path and exits with code 2. There is no flag to override (`--respect-user-dockerfile`), no env var (`THEOPACKS_ALLOW_USER_DOCKERFILE`), no warn-and-ignore branch.

**Rationale considered:**
- *Warn-and-ignore* — emits a warning and generates anyway. Two paths through the code; users rely on warnings being eventually-suppressed. Surprises users who expect their Dockerfile to be honored. FAANG-level systems don't have warnings for contract violations — they have errors.
- *Opt-in flag* — preserves the old path for "edge cases". Two paths means double the failure modes; the flag becomes load-bearing technical debt. No real edge case justifies it pre-release.
- *Hard fail* — one path. Contract is unambiguous. User must align (delete the file) or use a different tool. Errors are loud and actionable.

**Exit code 2** is conventional for "user-input invariant violated" (vs `1` which is generic failure). Distinguishes from "Docker daemon down" or "Go runtime crash" in CI logs.

**Consequences:**
- Enables: single contract surface; deterministic generation; eliminates the entire class of "buggy template Dockerfile blamed on theo-packs" misdiagnosis.
- Constrains: any user (none today) carrying a hand-tuned Dockerfile must delete it. They can still `docker build` outside theo-packs and deploy via a different mechanism — theo-packs simply won't be involved.

### D2 — Two coordinated PRs ship together; no staging
**Decision:** PR-A (theo-packs: hard fail + contract rewrite + test inversion) and PR-B (theo-stacks: delete 26 Dockerfiles + smoke test + READMEs) are merged on the same day. Neither is meaningful without the other:

- PR-A alone: theo-packs rejects all theo-stacks templates that still ship Dockerfiles → every deploy fails.
- PR-B alone: theo-stacks templates have no Dockerfile but theo-packs's old code path still tries to copy a non-existent file → cosmetic but the contract is unenforced.

**Rationale:** the two changes are a single semantic transition. Staging would create a window where the system is half-converted and worse than either end state.

**Consequences:**
- Enables: clean transition; one weekend's worth of merge coordination.
- Constrains: merge order matters across two repos. Recommended sequence: PR-A merges to `develop`, immediately PR-B merges to its `develop`, theo product picks up both via vendoring/submodule update or fresh clone.

### D3 — theo-stacks ships a smoke test that fails CI on Dockerfile presence
**Decision:** Add `tests/no_dockerfiles_test.sh` (or equivalent in whatever test framework theo-stacks uses) that runs `find templates -name Dockerfile -type f` and fails if the output is non-empty.

**Rationale considered:**
- *Just delete and trust reviewers* — works once. A future contributor copies a template from elsewhere with a Dockerfile, doesn't notice, ships. Contract regresses silently.
- *CI lint that scans templates* — overkill for one rule. A single shell-line `find` test is simpler and easier to maintain than a full lint config.
- *Pre-commit hook* — only catches local development. CI is the real gate.

**Consequences:**
- Enables: contract regression-proof at the team level.
- Constrains: theo-stacks's CI gains one trivial test. Doesn't extend to "no `.gitlab-ci.yml`" or other contract-adjacent rules; one rule per file is the convention.

### D4 — theo-stacks READMEs explicitly disclaim Dockerfiles, not just omit them
**Decision:** Each template's README gets a "Build" section with the text:

> ```markdown
> ## Build
>
> This template does not ship a Dockerfile. The Theo build pipeline runs
> [theo-packs](https://github.com/usetheodev/theo-packs) which detects the
> language/framework and generates an optimized Dockerfile at deploy time.
>
> If you fork this template and want to deploy outside Theo, you can:
> - Run `theopacks-generate` yourself to produce a Dockerfile, or
> - Write your own Dockerfile and use a non-theo-packs build path.
>
> Note that committing a Dockerfile inside this template's directory will
> cause `theopacks-generate` to **reject the build** (exit code 2). The
> contract is single-source-of-truth — one place generates Dockerfiles.
> ```

**Rationale:** absence is invisible. A new contributor cloning the template wonders "where's the Dockerfile?" and may add one to "fix" the gap. Explicit text shuts that down.

**Consequences:**
- Enables: every template explains the contract once.
- Constrains: 18 README updates (plus root README of theo-stacks). Boilerplate-heavy but mechanical.

### D5 — Contract document is rewritten, not patched
**Decision:** `docs/contracts/theo-packs-cli-contract.md` gets the "User-Dockerfile precedence" section deleted (not edited to say "removed"). A new "Single source of truth" preamble appears at the top, before "Scope". The Failure Modes table gains a row for "Dockerfile found in source tree → exit code 2".

**Rationale:** contracts that say "this used to be true; now it's not" age poorly. New readers ignore historical sections; old readers get confused about which is current. Clean rewrite > strikethrough.

**Consequences:**
- Enables: readable, current contract.
- Constrains: git history is the only record of the old contract. PR description preserves the old behavior verbatim for archaeology.

### D6 — CHANGELOG: `### Removed` AND `### Changed (BREAKING)` because of major-version implication
**Decision:** CHANGELOG entry under `[Unreleased]` includes:

```markdown
### Changed (BREAKING)
- theo-packs is now the single source of truth for Dockerfile generation. User-supplied
  Dockerfiles at `<source>/<app-path>/Dockerfile` cause the CLI to exit with code 2 and
  an error message. The previous "user-Dockerfile takes precedence" behavior has been
  removed entirely; there is no override flag.

### Removed
- `cmd/theopacks-generate/main.go` user-Dockerfile precedence path (lines 60-73 of the
  pre-removal file). See `docs/contracts/theo-packs-cli-contract.md` for the new contract.
- "User-Dockerfile precedence" section from `docs/contracts/theo-packs-cli-contract.md`.
- `TestUserProvidedDockerfileTakesPrecedence` from `cmd/theopacks-generate/main_test.go`
  (replaced by `TestUserProvidedDockerfileIsRejected`).
```

When theo-packs cuts its first non-pre-release version, this lands as a `[2.0.0]` major bump with the BREAKING marker carried forward.

**Rationale:** Keep a Changelog convention is `### Removed` for code/features; `### Changed` for behavior shifts. Both apply here. The `(BREAKING)` annotation is project convention for pre-release explicitness.

**Consequences:**
- Enables: discoverable history.
- Constrains: the next release is unambiguously a major version (`v2.0.0`), even if pre-release. Acceptable because theo-packs has never had a `v1.0.0` release tag — the bump is internal.

#### Deep Dives

**Replacement code:**
```go
// Single source of truth: theo-packs generates the Dockerfile. A user-
// supplied Dockerfile in the source tree is a contract violation — there
// would be two sources of truth for the build artifact, defeating the
// determinism theo-packs guarantees. Hard fail with exit code 2 (input
// invariant violated, distinguishable from generic failure code 1).
//
// See docs/contracts/theo-packs-cli-contract.md, section "Single source
// of truth", for the rationale.
userDockerfile := filepath.Join(appDir, "Dockerfile")
if _, err := os.Stat(userDockerfile); err == nil {
    fmt.Fprintf(os.Stderr,
        "[theopacks] ERROR: user-supplied Dockerfile found at %s.\n\n"+
            "theo-packs is the single source of truth for Dockerfile generation.\n"+
            "Remove the file and rerun. To opt out of generation entirely, do not\n"+
            "invoke theo-packs — declare your build via a different mechanism in your\n"+
            "deployment pipeline.\n",
        userDockerfile)
    os.Exit(2)
}
```

**Invariants:**
- After TA1.1, no code path in `theopacks-generate` reads or copies a user-supplied Dockerfile.
- Exit code 2 is reserved for this specific failure (no other code path uses it today; verified via `grep "os.Exit\|log.Fatal" cmd/theopacks-generate/main.go`).
- The error message names the path (actionable).

**Edge cases:**
- `os.Stat` returns an error other than `IsNotExist` (e.g., permission denied) — treat as "file does not exist, proceed". An opaque permission failure is not the user's fault and shouldn't block generation.
- Single-app project where `--app-path=.` so `appDir = source` and the Dockerfile would be at `<source>/Dockerfile` — same logic, same fail.
- Workspace where `--app-path=apps/api` so the check happens at `<source>/apps/api/Dockerfile` — same logic. A Dockerfile at the workspace root (`<source>/Dockerfile`) is NOT checked, because that's not under the app path. **This is intentional** — the workspace root may legitimately have a `Dockerfile` for local dev outside Theo (e.g., `docker compose up`). theo-packs only rejects within the app path it's analyzing.

func TestUserProvidedDockerfileIsRejected(t *testing.T) {
    // theo-packs is the single source of truth — user Dockerfiles cause
    // the CLI to exit with code 2 (D1 in the single-source-of-truth plan).
    bin := buildBinary(t)
    sourceDir := t.TempDir()

    require.NoError(t, os.WriteFile(
        filepath.Join(sourceDir, "Dockerfile"),
        []byte("FROM alpine:3.20\n"),
        0644,
    ))

    outputDir := t.TempDir()
    outputFile := filepath.Join(outputDir, "Dockerfile")

    cmd := exec.Command(bin,
        "--source", sourceDir,
        "--app-path", ".",
        "--app-name", "rejected-app",
        "--output", outputFile,
    )
    out, err := cmd.CombinedOutput()

    require.Error(t, err, "CLI must reject user-supplied Dockerfile")

    // Exit code 2 is the contract.
    var ee *exec.ExitError
    require.ErrorAs(t, err, &ee, "expected ExitError, got: %T", err)
    require.Equal(t, 2, ee.ExitCode(), "exit code must be 2 (input invariant violated)")

    // stderr names the path and explains.
    output := string(out)
    require.Contains(t, output, "user-supplied Dockerfile found at")
    require.Contains(t, output, filepath.Join(sourceDir, "Dockerfile"))
    require.Contains(t, output, "single source of truth")

    // No Dockerfile written to --output.
    _, statErr := os.Stat(outputFile)
    require.True(t, os.IsNotExist(statErr),
        "no Dockerfile should be written when the build is rejected")
}

func TestUserProvidedDockerfileAtWorkspaceRoot_IsNotRejected(t *testing.T) {
    // A Dockerfile at the workspace root (NOT inside the app path being
    // analyzed) is allowed — it may be there for local docker-compose
    // workflows unrelated to Theo. The contract only rejects within the
    // app path.
    bin := buildBinary(t)
    sourceDir := copyExampleToTemp(t, "node-npm")

    // Place a Dockerfile at <source>/Dockerfile but analyze a non-existent
    // sub-app path — wait, single-app projects use --app-path=.
    // So the workspace-root case is only meaningful for monorepos.
    // Use a synthetic monorepo: source/apps/api/* + source/Dockerfile.
    workspace := t.TempDir()
    require.NoError(t, os.MkdirAll(filepath.Join(workspace, "apps", "api"), 0755))
    require.NoError(t, os.WriteFile(
        filepath.Join(workspace, "package.json"),
        []byte(`{"name":"root","workspaces":["apps/*"]}`),
        0644,
    ))
    require.NoError(t, os.WriteFile(
        filepath.Join(workspace, "apps", "api", "package.json"),
        []byte(`{"name":"api","scripts":{"start":"node ."}}`),
        0644,
    ))
    require.NoError(t, os.WriteFile(
        filepath.Join(workspace, "Dockerfile"),  // workspace root, NOT inside apps/api
        []byte("FROM alpine:3.20\n"),
        0644,
    ))

    outputFile := filepath.Join(t.TempDir(), "Dockerfile.out")
    cmd := exec.Command(bin,
        "--source", workspace,
        "--app-path", "apps/api",
        "--app-name", "api",
        "--output", outputFile,
    )
